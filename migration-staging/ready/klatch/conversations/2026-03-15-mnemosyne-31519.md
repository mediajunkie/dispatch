## Claude Environment Bridging: Initial Research

### The landscape as it stands

There are currently **four official Claude access points** — claude.ai (browser), Claude Projects (browser), Claude Desktop (app), and Claude Code (terminal) — plus one new one that's highly relevant: **Cowork** (Desktop app, agentic mode). Each has a different relationship to your files, your memory, and your data portability.

The Torres article lays this out cleanly. Her comparison table's key axis is **portability**: Claude.ai and Projects store everything at Anthropic with no portability. Claude Code stores everything locally in markdown files on your machine — fully portable, no vendor lock-in. This is the fundamental tension Klatch is already navigating.

### The gap Klatch occupies

Klatch sits in a space none of these fill: a **local-first GUI** that talks to the API, owns its data in SQLite, and can import from multiple Claude environments. It's not a terminal tool (like Claude Code), not a cloud product (like claude.ai), and not a desktop agent (like Cowork). It's the orchestration layer.

### What's relevant to Klatch

**1. ClaudeSync — interesting but fragile**

ClaudeSync bridges local development environments with Claude.ai projects, enabling seamless synchronization. However, it's an independent, open-source project not affiliated with Anthropic, and users acknowledge potential violation of Anthropic's Terms of Service.

It's a one-way sync (local → Claude.ai Project), uses session key authentication scraped from the browser, and could break at any time. Two separate repos exist (jahwag and cshum forks) plus a VS Code extension. The TOS concern is real — this relies on undocumented APIs.

**Relevance to Klatch:** Low as a direct dependency (too fragile, TOS risk). But the *concept* is interesting: the idea that project knowledge files could be kept in sync between a local folder and a Claude.ai Project. If Anthropic ever made this official, it would directly serve Klatch's bridge-between-environments goal.

**2. Cowork — Claude Code for non-developers**

Anthropic released Cowork, a new AI agent capability that extends the power of its wildly successful Claude Code tool to non-technical users. It launched January 12, 2026 as a research preview for Max subscribers.

Key architecture details: Cowork is regular Claude Code wrapped in a less intimidating default interface and with a filesystem sandbox configured for you. Simon Willison discovered it uses VZVirtualMachine — the Apple Virtualization Framework — and downloads and boots a custom Linux root filesystem.

It can access local folders (sandboxed), read/edit/create files, use MCP connectors, and work with Claude in Chrome for browser automation. Cowork runs locally in an isolated virtual machine on your computer. It's now available on both Mac and Windows with full feature parity.

**Relevance to Klatch:** Medium-high as a **comparison point and potential integration surface**. Cowork can access local files — including, theoretically, Klatch's project folder. If Klatch generates markdown files (project context, export documents), Cowork could work with them. More importantly, Cowork's plugin/skill architecture is worth watching:

Plugins are the bridge between "general agent" and "reliable coworker." Each plugin bundles skills, connectors, slash commands, and sub-agent definitions into a single file-based package. The key word there is file-based: markdown and JSON, no code, no infrastructure, no build steps.

This is architecturally aligned with Klatch's values — file-based, auditable, local-first. A Klatch plugin for Cowork isn't crazy.

**3. The Claude Code ↔ Claude Projects gap — officially unfilled**

There's an open feature request to integrate Claude Code with Claude.ai Projects knowledge bases, enabling seamless knowledge integration, efficient caching, and a unified workflow. This was filed June 2025 and remains open.

Currently, Claude Code and Claude.ai Projects are **completely separate systems** with no native bridge. Claude Code uses local `CLAUDE.md` files and `~/.claude/` directory structure. Claude.ai Projects use uploaded knowledge files and cloud-stored instructions. There is no official way to connect them.

**Relevance to Klatch:** This is the gap Klatch is *already* partially filling. The import pipeline brings Claude Code sessions and claude.ai conversations into one place. The 4-layer prompt assembly reconstructs project context from both sources. Klatch is doing manually (via import) what the community is asking Anthropic to do natively.

**4. Torres comparison table — maps directly to Klatch's value proposition**

The table's dimensions (memory/context, file access, personalized shortcuts, portability) are exactly what Klatch addresses. Torres's core argument is that Claude Code's power comes from persistent local files as context, reusable commands, and parallel agents — all of which are either in Klatch already or on the roadmap (workflows, search).

The portability row is the killer: Claude Code stores all memory, context, and shortcuts locally on your machine — complete portability. If you switch to a different model provider, there's nothing to port. Klatch has the same property via SQLite.
